## 1. BMAD-METHOD

- **Version.** Stable **v6.8.0** (MIT license). Already installed in this repo at `_bmad/`
  (manifest: `_bmad/_config/manifest.yaml`, `installDate 2026-06-04`). Prerelease channel is
  `npx bmad-method@next install`. **We target v6.8.0 stable.**
  Source: https://github.com/bmad-code-org/BMAD-METHOD, `_bmad/_config/manifest.yaml`.

- **Custom module creation (BMB).** The BMad Builder module (`bmb`, v1.8.1) provides the
  `bmad-module-builder` skill with two actions: **Ideate Module** (`ideate-module`) and
  **Create Module** (`create-module`, "Scaffold module infrastructure into built skills, making
  them an installable BMad module"). Source: `_bmad/bmb/module-help.csv`.
  Per user decision we **hand-author** the module in the verified format rather than run the
  interactive builder, but the output shape mirrors what `create-module` produces.

- **Where modules live on disk / what survives an update.**
  - `bmad_builder_output_folder = "{project-root}/skills"` (`_bmad/bmb/config.yaml`). Built
    custom-module **source** goes in `{project-root}/skills/`. This folder is outside `_bmad/`
    and is **not** regenerated by the installer — it survives `bmad-method` updates.
  - The installer regenerates `_bmad/<module>/` (e.g. `_bmad/bmm/`, `_bmad/wds/`) on every
    install/update — **do not** keep durable custom code there.
  - `_bmad/custom/` holds only **config overrides** (`config.toml`, `config.user.toml`) that
    deep-merge over base config and survive updates. It is **not** for module code.
  - **Installed** skills live as a folder-per-skill under the IDE dirs (see §1 file format).
    These are regenerable install artifacts; re-running install restores them.
  Source: `_bmad/bmb/config.yaml`, `_bmad/custom/config.toml`, `_bmad/_config/manifest.yaml`.

- **File format (NOT YAML/TOML for the unit of work — it's Markdown skills).**
  - Top-level config: **TOML** (`_bmad/config.toml`, `_bmad/custom/config.toml`).
  - Per-module config: **YAML** (`_bmad/<module>/config.yaml`).
  - Module skill registry: **CSV** (`_bmad/<module>/module-help.csv`) with columns
    `module,skill,display-name,menu-code,description,action,args,phase,preceded-by,followed-by,required,output-location,outputs`.
  - The unit of work is a **skill**: a folder containing `SKILL.md` (Markdown + YAML frontmatter
    `name`, `description`), optional `customize.toml`, and for multi-step skills, step files
    (`steps-c/step-NN-*.md`) with frontmatter `name`/`description`/`nextStepFile`/`outputFile`.
  Source: `.claude/skills/bmad-create-story/`, `_bmad/wds/config.yaml`, `_bmad/wds/module-help.csv`.

- **Installed-skill locations per IDE** (all four kept in parity by the installer/sync):
  - `.claude/skills/<name>/SKILL.md`   — Claude Code (folder per skill) — **the one this harness uses**
  - `.agents/skills/<name>/SKILL.md`   — agents IDE (folder per skill)
  - `.zencoder/skills/<name>/SKILL.md` — zencoder (folder per skill)
  - `.opencode/commands/<name>.md`     — opencode (single flat file per command)
  Source: on-disk inspection (`ls .claude/skills`, `.agents/skills`, `.opencode/commands`).

- **Workflows = multi-step skills.** A workflow is a skill whose `SKILL.md` activates and then
  walks numbered step files. A step file invokes an agent by *instructing the running agent in
  prose* and chains via the `nextStepFile` frontmatter field; outputs are declared via
  `outputFile`. There is no separate machine "workflow engine" format — steps are Markdown read
  and executed by the harness agent. Source: `.claude/skills/bmad-testarch-test-design/steps-c/*`.

- **`project-context.md`.** Real. Generated by the `bmad-generate-project-context` skill into the
  module `output_folder` (`{project-root}/_bmad-output`); conventionally also referenced at
  `docs/project-context.md` / root. Loaded as a persistent fact by skills via
  `persistent_facts = ["file:{project-root}/**/project-context.md"]`.
  Source: `_bmad/bmm` module-help, `.claude/skills/bmad-create-story/customize.toml`.

## Deviations from the build plan (build against the real behavior)

1. **No `sm` agent.** The scrum-master / story-prep role the plan assigns to `sm` (steps 7–9) is
   provided by **skills**: `bmad-create-story`, `bmad-sprint-planning`, `bmad-dev-story`. Our story
   steps reference those skills, not a non-existent `sm` agent.
2. **Custom module code lives in `{project-root}/skills/`**, not `_bmad/custom/`. `_bmad/custom/` is
   config overrides only. We keep the `sdlc` module source in `skills/` and install copies into the
   IDE skill dirs; re-running install restores them after a BMAD update.
3. **Spec Kit and Impeccable are AI-harness slash-commands, not subprocess CLIs.** Only
   `npx repomix` and `npx impeccable detect` are true subprocesses. The future "glue" engine must
   orchestrate Spec Kit / Impeccable `document|extract|craft` by **driving the agent**, not by
   shelling out. (Relevant to later iterations; the first deliverable does not call these.)

## Phase 2 decisions (front-half build)

> Recorded 2026-06-06 while building front states 3/5/7 (`sdlc-author-architecture`,
> `sdlc-author-ui`, `sdlc-author-stories`) and reusing the one gate for all four reviews.

- **Contract representation = delimited surface block + SHA-256 lock.** `contract.md` wraps the shared
  cross-repo surface (API / events / data model, charter altitude) between the exact markers
  `<!-- CONTRACT-SURFACE:BEGIN -->` and `<!-- CONTRACT-SURFACE:END -->`. The SHA-256 of the bytes
  **between** the markers is stored in `epics/EP-<slug>/.sdlc/contract-lock.json` as
  `{ "artifact": "contract.md", "hash": "sha256:<hex>", "lockedAt": "<date>" }`. Recipe:
  `awk '/CONTRACT-SURFACE:BEGIN/{f=1;next} /CONTRACT-SURFACE:END/{f=0} f' contract.md | shasum -a 256`.
  **Verified 2026-06-06:** the hash round-trips (unchanged content → identical digest) and detects
  drift (any edit inside the block changes the digest). This is what the Phase 3 contract-check will
  diff against; a field-by-field structured diff is deferred to Phase 3. Full spec:
  `skills/sdlc-author-architecture/references/contract-format.md`.

- **Impeccable slash-command names confirmed** (already in §4, restated for the UI step): `/impeccable
  document` → `/impeccable extract` → `/impeccable craft` (existing project); `/impeccable craft` →
  `/impeccable extract` (new project). **Not installed in this repo as of 2026-06-06.** Decision:
  `sdlc-author-ui` documents these slash-commands and **degrades gracefully** — when Impeccable is
  absent the `ux-designer` lens authors `ui-design.md` + `DESIGN.md` directly and records
  `impeccable: not-installed` in the artifact frontmatter. No `npx impeccable skills install` is run in
  Phase 2 (tool install is out of the front-half scope).

- **Repomix interface = true CLI subprocess** (confirmed, unchanged from §3): `npx repomix@latest
  [flags]`, default output `repomix-output.xml`. It is **not** introduced in Phase 2 — backfill is
  Phase 3+. Recorded here per the Phase 2 build-plan requirement to confirm CLI-vs-slash-command:
  Repomix is a CLI (and `npx impeccable detect` is the only other true subprocess); Spec Kit and
  Impeccable `document|extract|craft` remain harness slash-commands (Deviation 3).
